## Findings and v0.1.4 corrections

| Area | v0.1.3 finding | v0.1.4 disposition |
| --- | --- | --- |
| HTTP error taxonomy | A real non-streaming `HTTPError` could relay the OpenAI-compatible `error` object without a top-level `codexhub_error`. | Preserve the upstream `error` object structure and add the typed `codexhub_error`; regression covers HTTP 429. |
| Gateway client authentication (`#24`) | The v0.1.3 gateway required its generated client key, but the Codex custom-provider overlay still sent OpenAI authentication, so a newly configured client could receive `401`. | Configure the managed Gateway provider with `requires_openai_auth = true` plus `experimental_bearer_token`. App CLI `0.144.0-alpha.4` verifies the local bearer still authenticates Gateway requests while the official account remains available; retain recognition of legacy CodexHub-managed providers. |
| Transport analyzer | Groups omitted `failure_class`, and retained only duration ranges rather than inspectable request/error examples. | Group by `failure_class`; retain bounded examples containing request ID, `content_length`, `duration_ms`, `error`, and `detail`. |
| Web Bridge | `config.origin` was inferred from `403` plus an error-message substring. | Use an explicit Rust error kind; backend text can no longer impersonate an origin-policy error. |
| Dead taxonomy branches | Python `backend_error` and `config_error` mappings existed only for unit-test inputs while the real Web Bridge was implemented in Rust. | Remove the unused Python classifications and test the real Rust paths. |
| Startup timeout test | A failure-path unit test waited five real seconds and also depended on a real health probe. | Inject the health probe and polling interval; deterministic timeout coverage completes below one second. |
| Release planning | A checked-in checkbox plan looked like a second live backlog. | Mark it as a historical snapshot and point all current status to GitHub Issues. |
| Provider page issue `#3` | The export-only explanation was already present in `v0.1.2`; reopening it would be incorrect. | Keep closed and retain its UI-contract coverage. |
| Version probe issue `#13` | The extension allowlist, timeout, and PATH tests are present and verified. | Keep/mark closed after verification. |

## Required human maintainer review

Before merging the v0.1.4 release PR into `main`, a human maintainer must explicitly confirm every item below in the PR review:

- [ ] The v0.1.3 findings above match the intended behavior and no reviewed change was silently dropped.
- [ ] Dynamic official catalog admission accepts Sol/Terra/Luna but still rejects unknown, disabled, denylisted, or forged entries.
- [ ] Unified-history repair never overwrites an unknown third-party `custom` provider and the off switch restores only ledger-owned official sessions.
- [ ] History synchronization never closes, restarts, or otherwise controls Codex; JSONL updates use snapshot verification plus atomic replacement, SQLite updates use a bounded busy timeout and short transaction, and concurrent writes defer without a completion receipt.
- [ ] The App-managed CLI JSONL evidence proves `spawn_agent -> wait -> close_agent`, one receiver thread, completed child state, and the expected sentinel for `ollama-cloud/glm-5.2`.
- [ ] `#10` and `#12` are closed only after their v0.1.4 fixes reach `main`; `#16` remains open for the post-release TLS/FlClash task.
- [ ] The complete v0.1.4 diff, updater manifest, installer signature, and `v0.1.2 -> v0.1.4` E2E evidence are acceptable.

**AI review is not a substitute for human maintainer approval.** AI or automated
review does not satisfy this gate. Until one human maintainer approves it, do not
merge to `main`, create the `v0.1.4` GitHub Release, or mark it Latest.
