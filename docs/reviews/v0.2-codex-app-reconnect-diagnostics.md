## Current Diagnostic Judgment

The latest real App probe separated two cases cleanly. Observed `[WinError 10053]` events after manual stop are downstream-client behavior and should not be used as evidence for an upstream OpenAI transport failure.

The actionable reconnect path is `request_id=15622e4ba31a`: Gateway had already relayed output text to Codex App, then the official upstream stream reset before any terminal SSE event. Because there was no upstream `response.completed` or `response.failed`, Codex App repeated the same prompt from the start.

The primary fix is to reduce the chance of that upstream reset. CodexHub's current official path uses Python `urllib` HTTPS to read a long-lived SSE stream. The real App evidence shows reset after an upstream silent gap of about `18.6s`, while direct official Codex App traffic does not reproduce the same way. That points at the CodexHub -> official upstream transport path, likely through Windows/VPN/TUN idle handling or HTTP stack differences.

Do not implement post-open replay for official passthrough. Once output deltas have been forwarded, replaying the same main-generation request is not idempotency-safe. If the upstream still disconnects, preserve the already forwarded bytes and terminate the stream explicitly.

## Added Official Transport Hardening

Official upstream HTTPS connections now install a `urllib` HTTPS handler that enables TCP keepalive on the raw socket before TLS wrapping:

- Windows: `SO_KEEPALIVE=1` plus `SIO_KEEPALIVE_VALS=(1, 5000ms, 5000ms)`.
- Other platforms: `SO_KEEPALIVE=1`, with `TCP_KEEPIDLE=5s`, `TCP_KEEPINTVL=5s`, and `TCP_KEEPCNT=3` where supported.
- Scope: only `upstream_name=official`; third-party providers stay on the default transport.
- Health feature marker: `official-upstream-tcp-keepalive`.

This is intended to keep the CodexHub -> official upstream TCP path active during model-generation silent gaps, preventing intermediaries from resetting a long SSE response around the 20-second idle mark.

Real App verification: the keepalive build completed a 231-second `gpt-5.5` streaming turn that previously reproduced the reset/restart pattern. This does not prove all official transport failures are eliminated; one side `gpt-5.4` request in the same run still hit a pre-response `request_write` reset and should be tracked separately from post-open stream-body disconnects.

## Added Passthrough Telemetry

Official passthrough streams now keep raw byte relay unchanged, but record a bounded SSE semantic summary on `request_complete` and `official_passthrough_stream_closed`:

- `sse_events_streamed`, `sse_json_events_streamed`, `sse_event_types`, `sse_event_type_counts`, and `sse_last_event_type`.
- `sse_terminal_event_seen`, `sse_completed_event_seen`, `sse_done_sentinel_seen`, `sse_response_event_seen`, and `sse_downstream_output_seen`.
- Multi-line SSE `data:` events are reassembled for telemetry parsing before JSON decode.
- Downstream write failures return telemetry `status=499`, `failure_class=downstream_client_closed`, and `client_disconnected=true`; upstream stream/read failures remain upstream-side failures.

Use these fields to decide whether a future `status=200` stream actually delivered `response.completed` and output-bearing events to the App before it repeats a turn.

## Added Stream Termination Fallback

When official passthrough fails while reading the upstream SSE body, Gateway now attempts to append a synthetic terminal Responses event before closing downstream:

- `event: response.failed` with `type=response.failed`, `response.status=failed`, and an upstream stream error payload.
- Telemetry fields `synthetic_terminal_event_sent`, `synthetic_terminal_event_type`, and optional `synthetic_terminal_write_error`.
- This is only attempted for `failure_side=upstream_read`; downstream write failures keep `synthetic_terminal_event_sent=false` because the client connection is already closed.

The expected Codex App behavior is to stop the current turn with an explicit failure instead of silently treating the stream as disconnected and reissuing the same prompt from the beginning.

## Node A/B Protocol

1. Capture Gateway health with `Invoke-RestMethod http://127.0.0.1:9099/health`.
2. On the suspected bad node, run one real Codex App probe turn and save the analyzer output.
3. Switch to the known good node.
4. Run the same real Codex App probe turn and save the analyzer output.
5. Compare groups by `failure_phase`, `failure_side`, `event`, `model_canonical`, `content_length` bucket, and `window_id`.

## Decision Criteria

Keep the current transport when the analyzer shows no recurrence in real App probe runs and failures correlate only with a bad external node that also breaks direct official traffic.

Add a per-process proxy setting when explicit `HTTP_PROXY` / `HTTPS_PROXY` materially improves real App probe recurrence without requiring global Windows proxy changes.

Evaluate an HTTP transport replacement when failures recur on a healthy node, direct official traffic remains stable, and analyzer groups concentrate in Gateway upstream open or stream body phases.

Do not implement post-open replay for official passthrough. Once downstream SSE has started, the local client has already observed part of the upstream stream.

Do not enable WebSocket for mixed-provider Gateway mode in v0.2. The mixed `custom` provider still has to advertise `supports_websockets=false`.

## External Audit Constraints

Related issues #17-#22 are follow-up constraints for interpreting reconnect evidence, not scope for this diagnostic branch.

- #17 retry idempotency: do not broaden main-generation retry behavior from this evidence unless upstream acceptance/idempotency is known.
- #18 downstream disconnect handling: keep `failure_side=upstream_read` separate from `failure_side=downstream_write` so client aborts do not look like upstream transport resets.
- #19 SSE reader queue/lifecycle: treat queue growth or reader lifetime as adjacent hardening symptoms; do not claim file descriptor leaks without reproduction.
- #20 multi-line SSE parsing: missing parsed upstream events can be parser loss when upstream sends multi-line `data:` events, so do not infer upstream silence solely from parsed-event absence.
- #21 concurrency limit: retry storms can amplify local saturation; analyzer output should separate upstream transport failures from provider capacity and local resource-pressure signals.
- #22 retry-storm dampening: use analyzer recurrence windows, failure phases, and attempt spacing to decide whether jitter, circuit breaker, or downstream notices should be pulled forward.
