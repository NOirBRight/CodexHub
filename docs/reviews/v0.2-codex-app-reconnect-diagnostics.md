## Current Diagnostic Judgment

The latest real App probe separated two cases cleanly. Observed `[WinError 10053]` events after manual stop are downstream-client behavior and should not be used as evidence for an upstream OpenAI transport failure.

The actionable reconnect path is `request_id=15622e4ba31a`: Gateway had already relayed output text to Codex App, then the official upstream stream reset before any terminal SSE event. Because there was no upstream `response.completed` or `response.failed`, Codex App repeated the same prompt from the start.

The July 9 diagnosis focused on reducing the chance of that upstream reset. At that time, CodexHub's official path used Python `urllib` HTTPS to read a long-lived SSE stream. The real App evidence showed a reset after an upstream silent gap of about `18.6s`, while direct official Codex App traffic did not reproduce the same way. That pointed at the CodexHub -> official upstream transport path, likely through Windows/VPN/TUN idle handling or HTTP stack differences.

Do not implement post-open replay for official passthrough. Once output deltas have been forwarded, replaying the same main-generation request is not idempotency-safe. If the upstream still disconnects, preserve the already forwarded bytes and terminate the stream explicitly.

## July 11 Recurrence And Transport Replacement

The July 9 TCP keepalive hardening did not resolve pre-response connection churn. In the 24-hour window ending `2026-07-11T13:45:19Z`, the Beta Gateway observed:

- 385 Official requests;
- 53 retry events affecting 42 requests (`10.9%`);
- 33 affected requests eventually recovered with `200`, while 9 ended in failure;
- 18 TLS handshake EOFs, 13 remote closes before a response, 11 request-write resets, 6 connect timeouts, and 5 local connection aborts.

All 53 retry events occurred before upstream response streaming began. Failures did not increase with Gateway concurrency: 29 of 198 requests that started with only one active Official request retried, compared with 11 of 132 at concurrency two and 2 of 41 at concurrency three. This rules out local request concurrency as the primary trigger for this incident window.

The previous `_OfficialKeepaliveHTTPSConnection` enabled TCP keepalive on each socket, but the cached `urllib` opener did not pool connections. `urllib.request` still opened a new TCP/TLS connection for every generation request. TCP keepalive therefore helped detect or prevent idle resets on an already-open stream but could not reduce the high rate of new handshakes that now dominate the telemetry.

The Official upstream path now uses a pinned, Gateway-owned `urllib3` connection pool:

- completed sequential requests reuse the same TCP/TLS connection;
- the pool is thread-safe, bounded to 16 connections, and blocks instead of creating unbounded connection bursts;
- streaming responses remain raw and non-preloaded;
- a fully consumed response returns its connection to the pool;
- a partial or failed stream closes and discards its connection;
- third-party Provider transport remains unchanged;
- existing Gateway retry classification and SSE relay remain outside the pool implementation.
- pooled sockets retain the July 9 TCP keepalive policy: Windows uses `SO_KEEPALIVE=1` plus
  `SIO_KEEPALIVE_VALS=(1, 5000ms, 5000ms)`; supported non-Windows platforms use a 5-second
  idle/interval and three probes.

Deterministic regression coverage proves that two sequential Official POST requests use one server-side TCP connection; the old `urllib` path used two. An isolated Gateway smoke using the App-managed Codex CLI completed ten consecutive `gpt-5.6-sol` turns with terminal SSE events and no Gateway retry. This is transport-layer evidence only; the real Desktop result below shows that it was not sufficient release evidence.

The superseded July 9 private-`urllib` opener first introduced that keepalive policy. A real App
verification completed a 231-second `gpt-5.5` streaming turn, but one side `gpt-5.4` request still
hit a pre-response `request_write` reset. The July 11 recurrence showed that keepalive alone was
insufficient because the opener did not reuse TCP/TLS connections; the pooled transport replaces
that opener rather than running alongside it. Its health feature marker is
`official-upstream-connection-pool`.

## July 11 Isolated Codex Desktop Result

A separate Codex Desktop process was launched with an isolated `CODEX_HOME`, isolated Chromium
data directory, and the current-source Gateway on `127.0.0.1:9113`. The existing Desktop process
and the user's Gateway on `9109` were not restarted or modified. The isolated task
`019f51a5-a0d0-7071-bd66-d8102cee4728` used `gpt-5.6-sol` and sequentially invoked the same
read-only command five times before returning a final answer.

The task produced six Official main-generation requests: five tool-decision rounds and one final
answer round. All six eventually returned `200` with an upstream `response.completed`, and Codex
Desktop wrote one final `agent_message` followed by `task_complete`. However, the third request
(`request_id=e36c9c7e78ab`) reproduced the release-blocking failure:

- the first attempt waited about 47 seconds without response headers;
- Gateway recorded `RemoteDisconnected: Remote end closed connection without response`;
- the event was classified as a pre-response `quick_transient` and retried after two seconds;
- the retry succeeded, but the request duration reached `57.758s` and the total Desktop task took
  `82.094s`;
- the other five main requests completed in `2.348-5.314s`;
- no post-output replay, duplicate final answer, or missing Desktop terminal event occurred.

An established Gateway-to-upstream connection observed after the retry was created at approximately
the retry time, and the remaining requests completed quickly. This is consistent with the replacement
connection being reused, but connection identity was not emitted by this build, so it is an inference
rather than direct per-request proof.

Two unrelated `400` responses from `gpt-5.4`/`gpt-5.4-mini` used different Desktop window ids and
are excluded from the six-request task result.

Conclusion at that checkpoint: pooling was working as a transport optimization, but it had not yet
eliminated the Official pre-response disconnect seen in the real Codex Desktop path, so Beta 10
remained blocked. The app-server reproduction below supplied the missing stale-connection and
post-terminal evidence.

## July 11 App-Server Reproduction And Fix

An agent-runnable probe now drives the App-managed `codex.exe app-server` over stdio and sends the
same `thread/start` and `turn/start` requests used by Desktop. Thirty small sequential turns completed
with 30 upstream `200` responses and no retry. Repeating the probe with the real Desktop session's
base instructions and dynamic tools, followed by ten separate shell calls, reproduced the failure:

- the first generation reused a connection that had been idle for about 116 seconds and received
  `RemoteDisconnected` before response headers;
- the retry returned a valid SSE stream containing `response.completed`;
- Gateway continued reading after that authoritative terminal event;
- 60 seconds later the HTTP body reader raised `IncompleteRead(0 bytes)`, so Gateway incorrectly
  appended a synthetic `response.failed` after the already-delivered `response.completed`;
- the complete App turn took 148.327 seconds.

The corrected transport now expires pooled Official connections after 30 idle seconds. Active tool
bursts still reuse their connection, while a human pause cannot hand a long-idle socket to the next
generation. After a terminal Responses SSE event, Gateway shortens the remaining HTTP drain window
to one second. A missing or malformed HTTP EOF after the terminal event is transport cleanup, not a
failed model response, and no synthetic failure is emitted.

The same real-session, ten-tool replay after deliberately idling the pool for 35 seconds completed in
33.639 seconds. All 11 generation requests returned `200`; there were zero retries, zero stream-close
events, and zero synthetic failures. The slowest generation took 3.975 seconds.

The current App-managed CLI also sends request bodies with `Content-Encoding: zstd`. The portable
Python runtime previously lacked the `zstandard` module and returned a local `400` before routing.
The runtime preparation step now downloads the pinned CPython 3.13 Windows wheel, verifies its
SHA-256, extracts it into the embeddable runtime, and treats importability as a readiness condition.

## Added Passthrough Telemetry

Official passthrough streams now keep raw byte relay unchanged, but record a bounded SSE semantic summary on `request_complete` and `official_passthrough_stream_closed`:

- `sse_events_streamed`, `sse_json_events_streamed`, `sse_event_types`, `sse_event_type_counts`, and `sse_last_event_type`.
- `sse_terminal_event_seen`, `sse_completed_event_seen`, `sse_done_sentinel_seen`, `sse_response_event_seen`, and `sse_downstream_output_seen`.
- Multi-line SSE `data:` events are reassembled for telemetry parsing before JSON decode.
- Downstream write failures return telemetry `status=499`, `failure_class=downstream_client_closed`, and `client_disconnected=true`; upstream stream/read failures remain upstream-side failures.

Use these fields to decide whether a future `status=200` stream actually delivered `response.completed` and output-bearing events to the App before it repeats a turn.

## Added Stream Termination Fallback

When official passthrough fails while reading the upstream SSE body, Gateway now attempts to append a synthetic terminal Responses event before closing downstream:

- `event: response.failed` with `type=response.failed`, `response.status=failed`, and an upstream stream error payload.
- Telemetry fields `synthetic_terminal_event_sent`, `synthetic_terminal_event_type`, and optional `synthetic_terminal_write_error`.
- This is only attempted for `failure_side=upstream_read`; downstream write failures keep `synthetic_terminal_event_sent=false` because the client connection is already closed.

The expected Codex App behavior is to stop the current turn with an explicit failure instead of silently treating the stream as disconnected and reissuing the same prompt from the beginning.

## Node A/B Protocol

1. Capture Gateway health with `Invoke-RestMethod http://127.0.0.1:9099/health`.
2. On the suspected bad node, run one real Codex App probe turn and save the analyzer output.
3. Switch to the known good node.
4. Run the same real Codex App probe turn and save the analyzer output.
5. Compare groups by `failure_phase`, `failure_side`, `event`, `model_canonical`, `content_length` bucket, and `window_id`.

## Decision Criteria

Keep the current transport when the analyzer shows no recurrence in real App probe runs and failures correlate only with a bad external node that also breaks direct official traffic.

Add a per-process proxy setting when explicit `HTTP_PROXY` / `HTTPS_PROXY` materially improves real App probe recurrence without requiring global Windows proxy changes.

Evaluate an HTTP transport replacement when failures recur on a healthy node, direct official traffic remains stable, and analyzer groups concentrate in Gateway upstream open or stream body phases.

Do not implement post-open replay for official passthrough. Once downstream SSE has started, the local client has already observed part of the upstream stream.

Do not enable WebSocket for mixed-provider Gateway mode in v0.2. The mixed `custom` provider still has to advertise `supports_websockets=false`; this closeout does not add WS or post-open replay.

For generic third-party provider transport instability and explicit-proxy A/B testing, see `docs/reviews/v0.2-generic-transport-instability.md`.

## External Audit Constraints

Related issues #17-#22 are follow-up constraints for interpreting reconnect evidence, not scope for this diagnostic branch.

- #17 retry idempotency: do not broaden main-generation retry behavior from this evidence unless upstream acceptance/idempotency is known.
- #18 downstream disconnect handling: keep `failure_side=upstream_read` separate from `failure_side=downstream_write` so client aborts do not look like upstream transport resets.
- #19 SSE reader queue/lifecycle: treat queue growth or reader lifetime as adjacent hardening symptoms; do not claim file descriptor leaks without reproduction.
- #20 multi-line SSE parsing: missing parsed upstream events can be parser loss when upstream sends multi-line `data:` events, so do not infer upstream silence solely from parsed-event absence.
- #21 concurrency limit: retry storms can amplify local saturation; analyzer output should separate upstream transport failures from provider capacity and local resource-pressure signals.
- #22 retry-storm dampening: use analyzer recurrence windows, failure phases, and attempt spacing to decide whether jitter, circuit breaker, or downstream notices should be pulled forward.
