# Gateway Upstream Transport Diagnostics (#16)

Date: 2026-07-12

## Scope and Evidence Boundary

This note covers the diagnostic/evidence slice for generic Gateway upstream transport failures. It does not change Gateway transport behavior, Python proxy settings, or protocol translation.

The historical third-party observation involved `opencode` and an Ollama Cloud model with TLS EOF and connect-timeout errors. It is useful for testing whether the analyzer separates provider paths, but it is not evidence for the official Codex App reconnect issue. The official product decision below is driven only by the incident evidence that direct official Codex traffic failed on the same bad node and recovered after a node switch.

The checked-in JSONL fixture is entirely synthetic. It contains no account, session, thread, local-path, or production request identifiers and is a regression input, not a claim of a fresh reproduction.

## Analyzer

Analyze a captured telemetry JSONL file with an explicit incident-bucket size:

```powershell
python scripts/analyze_transport_failures.py --input <telemetry-jsonl> --since <inclusive-utc-timestamp> --until <inclusive-utc-timestamp> --window-minutes 5
```

The report separates `provider_scope=official` from `provider_scope=third_party` and groups failure events by:

- provider and canonical model
- client
- event, failure phase, and failure side
- error and failure class
- request-size bucket
- deterministic UTC `time_window`

`time_window` replaces telemetry correlation IDs as a report dimension, so multiple requests in the same incident bucket aggregate instead of fragmenting by task or session context. Explicit runtime `failure_class` values take precedence. For older records that lack one, the analyzer labels only evidence-backed legacy patterns: `tls_eof`, `connect_timeout`, and `connection_reset`; other cases remain `unknown`. `connect_timeout` requires connect-specific evidence such as `WinError 10060` or an explicit connect attempt; an ambiguous read or stream timeout stays `unknown` rather than being mislabelled as a connection failure.

An official `official_passthrough_stream_closed` record with `failure_side=downstream_write`, `failure_class=downstream_client_closed`, and `status=499` is a downstream cancellation, not upstream transport evidence. It is excluded from `failure_count` and all A/B groups; `excluded_downstream_cancellation_count` reports it separately without inflating upstream totals.

When `--since` or `--until` is supplied, the analyzer compares parsed UTC timestamps rather than raw strings. It reports skipped failure counts for out-of-window, missing-timestamp, and invalid-timestamp records so an A/B comparison cannot silently include an ambiguous boundary record.

Run the checked-in regression input:

```powershell
python scripts/analyze_transport_failures.py --input tests/fixtures/transport_failures.jsonl --window-minutes 30
```

The expected synthetic report contains separate official stream-reset/TLS-EOF groups and third-party TLS-EOF/connect-timeout groups, plus one separately counted downstream cancellation that does not enter the upstream groups.

## Authorized Local A/B Checklist

No live A/B run was performed for this branch. The task explicitly prohibits changing shared Gateway configuration, restarting shared apps, or changing live proxy routing without Orchestrator authorization. Run this checklist only in an authorized, isolated local session.

### Controls shared by both conditions

1. Record the Gateway commit, UTC start/end, proxy-node label, provider, canonical model, client, prompt/workload identifier, iteration count, and whether direct official traffic is healthy. Do not record credentials, account identifiers, or raw task/session identifiers.
2. Use the same provider/model/client workload, request count, concurrency, and proxy node for both conditions. Run each condition on a separate unused loopback port; do not restart or reconfigure the shared Gateway.
3. Capture only the resulting local telemetry JSONL and analyzer output. Redact identifiers before attaching evidence to an issue or PR.
4. Compare both runs with the same bounds and bucket size, including `provider_scope`, provider/model/client, `failure_phase`, `failure_side`, `failure_class`, `error`, `time_window`, and failures per recorded workload count.

### A. Transparent/TUN route

In a dedicated terminal, clear only that process's explicit proxy variables before starting an isolated Gateway:

```powershell
Remove-Item Env:HTTP_PROXY, Env:HTTPS_PROXY -ErrorAction SilentlyContinue
$gatewayScript = Join-Path (Get-Location) "src-python/codex_proxy.py"
python $gatewayScript --host 127.0.0.1 --port <transparent-port>
```

Supply any existing local Gateway client key only in the terminal session; do not save it in scripts, fixtures, or captured output. Run the fixed workload through `<transparent-port>`, then save the analyzer JSON.

### B. Explicit HTTP CONNECT proxy route

In a separate dedicated terminal, set only process-local proxy variables and start another isolated Gateway:

```powershell
$env:HTTP_PROXY = "http://127.0.0.1:7890"
$env:HTTPS_PROXY = "http://127.0.0.1:7890"
$gatewayScript = Join-Path (Get-Location) "src-python/codex_proxy.py"
python $gatewayScript --host 127.0.0.1 --port <explicit-proxy-port>
```

Run the identical workload through `<explicit-proxy-port>`, then analyze it with the same `--since`, `--until`, and `--window-minutes` values. Clear the terminal's environment or close the terminal after the run; do not change Windows system or WinHTTP proxy settings.

## Recommended Product Path

Keep the current Gateway transport behavior for now; do not add an app-managed per-process proxy setting or replace `urllib` from this evidence slice.

This is the least speculative path because the official failure coincided with direct official Codex App stream failures on the same node and recovered after the node changed. That supports a network/proxy-node contribution more strongly than a Gateway-only transport defect. The analyzer now gives a bounded way to test the competing routing hypothesis without conflating official and third-party failures.

Uncertainty remains material: the third-party record is historical and not causal evidence for the official incident, the fixture is synthetic, and no authorized controlled A/B has yet been run. Reconsider the recommendation only if an authorized same-workload A/B materially lowers recurrence with explicit proxy routing, or if Gateway-only failures persist on a healthy node while direct upstream traffic remains stable. Those outcomes would respectively justify a per-process proxy setting or a focused transport-implementation investigation.
