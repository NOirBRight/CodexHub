## Local A/B Checklist

The open question is whether Gateway's Python process should use transparent/TUN routing or an explicit per-process HTTP CONNECT proxy for upstream calls.

### A. Transparent/TUN routing

```powershell
Remove-Item Env:HTTP_PROXY, Env:HTTPS_PROXY -ErrorAction SilentlyContinue
$env:CODEX_PROXY_GATEWAY_CLIENT_KEY="codexhub-proxy"
python D:\Workstation\CodexHub\src-python\codex_proxy.py --host 127.0.0.1 --port 9100
```

Run the same third-party provider workload through `127.0.0.1:9100`, then analyze the resulting telemetry JSONL.

### B. Explicit proxy routing

```powershell
$env:HTTP_PROXY="http://127.0.0.1:7890"
$env:HTTPS_PROXY="http://127.0.0.1:7890"
$env:CODEX_PROXY_GATEWAY_CLIENT_KEY="codexhub-proxy"
python D:\Workstation\CodexHub\src-python\codex_proxy.py --host 127.0.0.1 --port 9101
```

Run the same provider/model/client workload through `127.0.0.1:9101`, then compare analyzer groups by:

- `provider_scope`
- `provider_id`
- `model_canonical`
- `client_id`
- `failure_phase`
- `failure_side`
- `error`
- request size bucket
- `window_id`

## 2026-07-10 Reproducibility Note

Not reproduced on 2026-07-10. No fresh local third-party provider A/B run was performed in this closeout branch. The 0.1.3 scope is diagnostic/reporting closeout, not a transport behavior change.

## Product Decision for 0.1.3

Keep the current transport behavior for 0.1.3.

Do not enable WebSocket for mixed-provider Gateway mode in this release.

Open a follow-up app-managed per-process proxy issue only if the A/B checklist shows that explicit `HTTP_PROXY` / `HTTPS_PROXY` materially reduces recurrence compared with transparent/TUN routing for the same provider/model/client workload.

Evaluate replacing Python `urllib` only if failures recur with explicit proxy enabled, direct provider traffic is stable, and analyzer groups concentrate in Gateway upstream-open or stream-body phases.
