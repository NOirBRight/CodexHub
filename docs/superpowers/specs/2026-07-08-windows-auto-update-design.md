# Windows Auto Update Design

## Goal

CodexHub v0.1.0 Free Beta should ship as a Windows-only GitHub Releases installer with product-visible version management and a safe auto-update path. The release remains free and unsigned at the Windows Authenticode layer, but every updater package must be verified by Tauri's updater signature before installation.

The user experience target is simple:

- Users install CodexHub from GitHub Releases.
- CodexHub shows its current version in settings.
- CodexHub checks for updates on startup and by manual user action.
- CodexHub only installs an update after the user confirms.
- Windows SmartScreen warnings are documented as an expected unsigned beta behavior.

## Sources

- Tauri updater documentation: `https://v2.tauri.app/plugin/updater/`
- Tauri updater JavaScript API reference: `https://v2.tauri.app/reference/javascript/updater/`
- CodexHub repository remote: `https://github.com/NOirBRight/CodexHub.git`

## Current State

CodexHub is already a Tauri 2 desktop app with a React/Vite frontend and Rust backend commands exposed through `frontend/src/lib/tauri.ts`. The current settings entry point is `frontend/src/components/SettingsDrawer.tsx`; `frontend/src/pages/SettingsPage.tsx` is not mounted by `frontend/src/App.tsx`.

Release packaging is not yet enabled:

- `src-tauri/tauri.conf.json` has `bundle.active = false`.
- There is no Tauri updater plugin dependency or updater configuration.
- There are no app update commands in `src-tauri/src/main.rs`.
- The frontend does not expose app version, update availability, or update installation actions.

## Decisions

### Distribution

The first public package is Windows x64 only and distributed through GitHub Releases. The preferred installer artifact is the NSIS setup executable generated by Tauri.

The updater endpoint is the latest-release static JSON file:

```text
https://github.com/NOirBRight/CodexHub/releases/latest/download/latest.json
```

The static JSON format must include a SemVer-compatible version, release notes, a Windows x64 package URL, and the package signature generated for that build.

### Signing

CodexHub v0.1.0 uses Tauri updater signing only.

- The updater public key is committed in `src-tauri/tauri.conf.json`.
- The updater private key is stored outside the repo and injected into CI as a secret.
- Windows Authenticode signing is not part of v0.1.0.
- SHA256 checksums are produced for release assets as a user-verifiable fallback.

### Update Behavior

CodexHub performs a delayed startup update check after the first runtime load. Startup checks are best-effort:

- If no update is available, the app stays silent.
- If an update is available, the app shows a toast with the new version and an install action.
- If the check fails, the app stays silent unless the user manually triggered the check.

Manual update checks live in the settings drawer:

- Current version is always visible.
- "Check for updates" reports no update, update available, or check failure.
- "Install update" appears only when an update is available.
- Installation requires user action and clearly states that CodexHub will close while the installer runs.

No background forced installation is allowed in v0.1.0.

### Settings Drawer Placement

The update UI must be placed in the existing `SettingsDrawer`, not in `SettingsPage` and not on the Gateway page.

Exact placement:

- Opened from the top runtime bar settings button.
- Inside the first `CodexHub` settings section.
- Inside that section's existing rounded settings panel.
- Immediately below the language selector.
- Above the auto-start, official-models, unified-history, bound-client-sync, and repair-history controls.

The drawer order should be:

1. Header: `Settings` / `CodexHub & Gateway`
2. `CodexHub` section
3. Language selector
4. `Version & Updates` block
5. CodexHub behavior toggles and repair-history action
6. Auto retry section
7. Image proxy section
8. Save footer

The update block is part of the CodexHub product settings, not part of Gateway runtime settings. This keeps app lifecycle and version controls visible without scrolling past Gateway-specific options.

## Backend Design

Add `src-tauri/src/app_updates.rs` to own desktop update behavior. The module exposes Tauri commands through `main.rs`:

- `get_app_version(app: AppHandle) -> AppVersionInfo`
- `check_app_update(app: AppHandle) -> AppUpdateStatus`
- `install_app_update(app: AppHandle) -> AppUpdateInstallResult`

The returned TypeScript shape is mirrored by `frontend/src/lib/types.ts`.

```rust
pub struct AppVersionInfo {
    pub current_version: String,
}

pub struct AppUpdateStatus {
    pub available: bool,
    pub current_version: String,
    pub latest_version: Option<String>,
    pub checked_at: String,
    pub notes: Option<String>,
    pub date: Option<String>,
}

pub struct AppUpdateInstallResult {
    pub installed: bool,
    pub version: String,
    pub message: String,
}
```

`check_app_update` calls `app.updater()?.check().await`. A `None` response means no update is available. An update response is mapped to `AppUpdateStatus`.

`install_app_update` checks again, downloads, and installs the update through Tauri. On Windows, the app is expected to exit during installation. If no update is available by the time installation starts, the command returns `installed = false` with the current version.

The module should keep error messages concrete and user-facing:

- Missing updater configuration: "App updates are not configured in this build."
- Network or GitHub metadata failure: "Failed to check for updates: ..."
- Download or install failure: "Failed to install update: ..."

## Tauri Configuration

Update `src-tauri/Cargo.toml`:

- Add `tauri-plugin-updater = "2"`.

Update `src-tauri/src/main.rs`:

- Add `mod app_updates;`.
- Register `tauri_plugin_updater::Builder::new().build()`.
- Register the three update commands in `tauri::generate_handler!`.

Update `src-tauri/tauri.conf.json`:

- Set `bundle.active = true`.
- Set `bundle.targets = ["nsis"]`.
- Set `bundle.createUpdaterArtifacts = true`.
- Add `plugins.updater.pubkey`.
- Add `plugins.updater.endpoints` with the GitHub Releases `latest.json` URL.
- Set Windows updater install mode to `passive`.

The implementation must not commit the updater private key.

## Frontend Design

Update `frontend/src/lib/types.ts` with:

- `AppVersionInfo`
- `AppUpdateStatus`
- `AppUpdateInstallResult`

Update `frontend/src/lib/tauri.ts` with desktop-only API wrappers:

- `getAppVersion`
- `checkAppUpdate`
- `installAppUpdate`

These wrappers use `desktopCall`, not `call`, because browser preview and the local web bridge cannot perform desktop update installation. In browser preview, they return `null`.

Update `frontend/src/components/SettingsDrawer.tsx`:

- Add a `Version & Updates` block inside the existing `CodexHub` section's rounded settings panel.
- Place the block immediately below the language selector and above the CodexHub behavior toggles.
- Show the current version.
- Show a check button.
- Show update availability details when present.
- Show an install button only after an update is available.
- Disable update buttons while a check or installation is in progress.
- Do not add update controls to `SettingsPage`, because it is not mounted by the current app shell.

Update `frontend/src/App.tsx`:

- Run one delayed startup update check after the initial runtime load.
- Suppress startup update-check errors.
- Show a toast only when an update is available.
- Toast action invokes the same install path used by Settings.

Update `frontend/src/i18n/locales/zh-CN.ts` and `frontend/src/i18n/locales/en-US.ts` with all user-visible update strings.

## Error Handling

Manual checks are explicit user actions and should show errors.

Startup checks are background convenience and should not show errors. This prevents GitHub outage, missing prerelease metadata, local firewall issues, or development builds from creating noisy startup banners.

If update installation starts, the UI should show an installing toast and avoid claiming success after the app exits. If the command returns with an error, the toast becomes an error.

## Testing

Backend tests should cover pure mapping and version behavior where practical:

- App version command returns the package version.
- No-update responses map to `available = false`.
- Update metadata maps to `available = true`, latest version, notes, and date.
- User-facing errors include the failing operation.

Frontend contract tests should cover:

- Update APIs exist in `frontend/src/lib/tauri.ts`.
- Settings drawer renders current version and update controls.
- Startup update check suppresses errors and only toasts on availability.
- Browser preview treats update APIs as desktop-only.
- English and Chinese locales contain update strings.

Release verification should cover:

- `npm run build` in `frontend`.
- `npm run test:ui-contract` in `frontend`.
- `cargo test` in `src-tauri`.
- Tauri Windows build produces an NSIS installer.
- Updater artifacts and `.sig` files are generated.
- `latest.json` validates against Tauri's static JSON format.
- A locally hosted update manifest can update from one test version to a newer test version.

## Out of Scope

The v0.1.0 update work does not include:

- Windows Authenticode signing.
- Forced updates.
- Silent updates.
- Rollbacks or downgrades.
- Multiple update channels.
- A custom update server.
- macOS or Linux installers.
